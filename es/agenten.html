El agente individual

El Agente AIMOS

Especialistas autónomos con memoria propia, herramientas propias y personalidad — cada agente es un asistente digital especializado.

Fundamentos

¿Qué es un agente AIMOS?

Un agente AIMOS es un asistente digital especializado que combina cuatro componentes clave:

M

Memory (Memoria)

Base de datos SQLite propia con memoria a largo plazo semántica, episódica y procedimental.

S

Skills (Herramientas)

Colección de herramientas especializadas: consultas SQL, llamadas REST, operaciones de archivos, funciones específicas del dominio.

L

LLM (Modelo de lenguaje)

Modelo local mediante runtime LLM local. El agente construye el prompt completo a partir del system prompt, contexto de memoria y consulta del usuario.

C

Connector (Conexión)

Canal de comunicación con el usuario: Telegram, e-mail, voz o dashboard.

Modo de trabajo

El agente como asistente digital

Un agente AIMOS no trabaja como un chatbot — trabaja como un empleado administrativo con su propio escritorio, su propio archivo y su propio ritmo diario.

Memoria

Recuerda hechos, correcciones, preferencias — de forma permanente. La búsqueda semántica encuentra recuerdos incluso con otras palabras.

Notas y archivo

Redacta resúmenes, resultados de investigación y notas de conversación en su propio workspace — como un empleado que mantiene su archivo.

Calendario

Citas, plazos, seguimientos. El agente revisa en cada interacción sus citas pendientes y vencidas — y actúa de forma proactiva.

Lista de tareas

Tareas pendientes, delegaciones, listas de espera. Se actualiza automáticamente en el ciclo Dreaming.

Ritmo diario

Una vez al día, el agente revisa de forma autónoma el calendario, los correos electrónicos y las tareas pendientes — sin que nadie le dirija la palabra.

Dreaming

En períodos de inactividad, el agente procesa sus conversaciones: extrae hechos, actualiza notas y listas de tareas, consolida su memoria.

Trabajo en equipo

Los agentes se comunican entre sí — el agente financiero pregunta al agente de logística por los valores de inventario, sin molestar al usuario.

Escalado

Ante preguntas complejas, el agente escala automáticamente a un modelo de IA más potente — de forma anonimizada y con registro auditable.

El punto clave: El agente prepara — usted decide. Su conocimiento especializado, su experiencia y su criterio siguen siendo indispensables. El agente le quita el trabajo rutinario: recopilar datos, supervisar plazos, preparar borradores. El trabajo profesional sigue siendo suyo.

Conexión

Conectores disponibles actualmente

El agente se comunica a través de conectores — interfaces estandarizadas hacia usuarios, sistemas y otros agentes. Se desarrollan nuevos conectores continuamente y pueden añadirse en cualquier momento para su infraestructura de TI específica.

Telegram

Texto, mensajes de voz, documentos. Mensajes proactivos para recordatorios, alarmas y resultados. Shared Listener para todos los agentes.

E-Mail

IMAP/SMTP para envío y recepción. Monitorización POP3 para buzones entrantes. Formato HTML y archivos adjuntos.

Voice

Whisper STT + Piper TTS — completamente local. Reconocimiento y síntesis de voz en todos los idiomas, sin servicios en la nube.

SFTP

Acceso a archivos en equipos de trabajo vía Tailscale VPN. Carpeta compartida para DXF, PDF, Excel — cifrada y sin puertos abiertos.

Bases de datos SQL

PostgreSQL, MSSQL, Firebird — exclusivamente consultas SELECT. Sin acceso de escritura a datos de producción. Read-Only by Design.

REST / SOAP API

Conexión API universal para ERP, CRM, gestión de mercancías. GET, POST, PUT con autenticación configurable.

Arquitectura de Threads

Un agente, muchos clientes — simultáneamente

Cada conversación recibe su propio ID de thread. El agente solo ve al cliente actual — sin importar cuántos se ejecuten en paralelo.

Threads Aislados

Cada cliente recibe automáticamente su propio thread. El usuario de Telegram A nunca ve la conversación del cliente de correo B — aunque el mismo agente atienda a ambos.

Multicanal

Un cliente escribe por Telegram: «Le envié un correo.» El agente encuentra el thread de correo, verifica la identidad y tiene el contexto de inmediato.

Propagación de Thread

Cuando un agente delega una tarea a un colega, el ID de thread viaja con ella. El destinatario trabaja en el mismo contexto de cliente — sin riesgo de confusión.

Ejemplo: Tres clientes, un equipo de agentes

Thread tg:4819

Cliente Müller (Telegram)

→ Soporte: Asesoría

→ Back Office: Análisis de imagen

→ Soporte: Resultado al cliente

Thread email:schmidt@...:a3f2

Cliente Schmidt (Correo)

→ Soporte: Leer consulta

→ Administración: Crear presupuesto

→ Soporte: Reenviar confirmación

Thread tg:7291

Cliente Weber (Telegram)

→ Soporte: Pregunta de mantenimiento

→ Base de conocimiento consultada

→ Soporte: Respuesta directa

Los tres procesos se ejecutan en paralelo. Ningún agente ve los datos de otro cliente.

Asignación automática de thread

Telegram, correo, panel de control — cada canal genera automáticamente el ID de thread correcto al recibir un mensaje.

Threading de correo

Las respuestas a correos existentes se asignan al thread correcto mediante las cabeceras In-Reply-To y References.

Archivos por thread

Fotos, documentos y adjuntos se almacenan por thread — sin importar si llegan por Telegram o correo electrónico.

Aislamiento a nivel de código

La separación se aplica a nivel de base de datos, no depende del modelo de IA. Incluso un modelo defectuoso no puede mezclar datos de clientes.

3-Tier Memory

Memoria a largo plazo

Tres tipos de memoria, búsqueda híbrida y un ciclo Dreaming para la consolidación.

Dreaming — Consolidación de memoria por llamada LLM

En estado de reposo, el agente analiza sus conversaciones con una llamada LLM, extrae hechos, actualiza notas y listas de tareas, consolida su memoria y genera informes semanales.

Como el cerebro humano durante el sueño — el agente condensa experiencias en conocimiento, elimina entradas redundantes y refuerza conexiones importantes. El resultado: respuestas más precisas con menor consumo de tokens.

Modelo de lenguaje

El LLM — cerebro del agente

El Large Language Model (LLM) es el motor de pensamiento detrás de cada agente. Comprende el lenguaje, toma decisiones y controla herramientas — y funciona completamente en su propio hardware.

¿Qué hace el LLM?

  • Comprende consultas — en lenguaje natural, en cualquier idioma
  • Selecciona herramientas — decide de forma autónoma si se necesita una consulta a base de datos, un envío de correo o un cálculo
  • Formula respuestas — técnicamente correctas, en el contexto de la conversación previa
  • Aprende de correcciones — a través de la memoria a largo plazo, no mediante entrenamiento del modelo

¿Por qué local en lugar de la nube?

Soberanía de datos

Sus consultas nunca abandonan la red. Ningún proveedor de nube ve sus datos.

Sin costes recurrentes

Sin precio por token por consulta. El modelo funciona de forma ilimitada en su GPU.

Disponibilidad

Sin límite de API, sin rate-limiting, sin dependencia de servicios externos.
