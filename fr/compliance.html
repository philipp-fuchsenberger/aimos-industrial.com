Piste d'audit

Chaque action d'agent est journalisée sans lacune. La piste d'audit capture six types d'événements :

SYSTEM_PROMPT Prompt système complet pour la traçabilité
REQUEST Requête utilisateur entrante
RESPONSE Réponse complète de l'agent
TOOL_START Appel d'outil avec paramètres
TOOL_OK Résultat de l'outil (succès ou erreur)
LLM_USAGE Consommation de tokens, modèle, latence

Protection des données UE

RGPD / GDPR

AIMOS ne se conforme pas au RGPD par adaptation rétroactive, mais par des décisions architecturales — Privacy by Design.

Art. 25 — Protection des données dès la conception

  • Traitement des données local par défaut (SovereignNode)
  • PII-Vault anonymise avant toute transmission externe
  • Aucun service cloud dans la configuration standard
  • Minimisation des données : seules les données contextuelles pertinentes dans le prompt LLM

Art. 32 — Sécurité du traitement

  • Chiffrement : PostgreSQL SSL, SQLite on-disk
  • Contrôle d'accès via Execution Rings (3 niveaux)
  • Piste d'audit complète de toutes les opérations de traitement
  • Mappings PII liés à la session (pas de stockage PII persistant)

Conformité aux normes

ISO 9001 & 27001

AIMOS répond aux exigences des deux normes grâce à des mécanismes intégrés.

ISO 9001 — Management de la qualité

  • Traçabilité : chaque décision dans le journal d'audit
  • Chaîne de traçabilité : les Session-IDs relient tous les événements
  • Processus documentés : les prompts système comme procédures opérationnelles
  • Actions correctives : le cycle Dreaming apprend des erreurs

ISO 27001 — Sécurité de l'information

  • Classification : les Execution Rings définissent les niveaux d'accès
  • Contrôle d'accès : activation des outils configurable par agent
  • Journalisation des événements : les 6 types d'événements capturés
  • Sécurité physique : les données ne quittent pas le SovereignNode

Comptabilité

Conformité GoBD

La GoBD (principes pour la tenue et la conservation régulières des livres, registres et documents sous forme électronique) impose des exigences particulières aux systèmes assistés par IA.

AIMOS résout cela par une séparation architecturale stricte :

Séparation : brouillon vs. validation

Les agents préparent les écritures comptables (brouillon). La validation est effectuée exclusivement par des comptables autorisés dans le système cible (ETA V8, DATEV, etc.). Aucun LLM n'a de droits de transaction.

Human-in-the-Loop

Les actions Ring 2 (accès en écriture) nécessitent une activation explicite. Les opérations comptables nécessitent toujours une approbation humaine — imposé par l'architecture, pas par une politique.

Réglementation UE

EU AI Act (Règlement IA)

Le règlement IA de l'UE (UE) 2024/1689 entre progressivement en vigueur et concerne toute entreprise utilisant des systèmes d'IA. AIMOS est architecturalement conçu pour la conformité — pas adapté après coup.

Classification des risques : risque limité

Les agents AIMOS sont des systèmes d'assistance avec supervision humaine (Human-in-the-Loop). Pas de décisions autonomes concernant des personnes, pas de surveillance biométrique, pas de fonctions de scoring social. Ils relèvent donc de la catégorie « risque limité » — pas « haut risque ».

Obligation de transparence : respectée

Art. 50 : les utilisateurs doivent savoir qu'ils interagissent avec une IA. Chaque agent AIMOS s'identifie dans son prompt système comme assistant IA. Toutes les actions sont traçables dans la piste d'audit.

Supervision humaine : imposée par l'architecture

Art. 14 : les Execution Rings garantissent qu'aucun agent n'exécute de manière autonome des actions critiques. Les écritures comptables, e-mails et commandes sont présentés comme brouillons — l'humain décide.

Documentation technique : complète

Art. 11 : architecture système, flux de données, description du modèle, indicateurs de performance et évaluation des risques sont documentés et consultables. Licence MIT : accès complet au code source.

Journalisation : sans lacune

Art. 12 : chaque action d'agent est journalisée — appels d'outils, requêtes LLM, consommation de tokens, chaînes de décision. Pistes d'audit révisables avec Session-IDs et horodatages.

Souveraineté des données : complète

Art. 10 : les données d'entraînement et de traitement ne quittent pas le SovereignNode. Le PII-Vault anonymise automatiquement les appels API externes. Aucune dépendance cloud.

Avantage de l'IA locale

Les services d'IA cloud doivent se conformer à des obligations plus strictes en tant que « IA à usage général » (GPAI). AIMOS utilise un modèle open source exploité localement — les obligations GPAI incombent au fournisseur du modèle (Alibaba/Qwen), pas à l'opérateur.
