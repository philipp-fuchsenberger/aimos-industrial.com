Journalisation

Piste d'audit

Chaque action d'agent est journalisée de manière exhaustive. La piste d'audit capture six types d'événements :

SYSTEM_PROMPT System-Prompt complet pour la traçabilité
REQUEST Requête utilisateur entrante
RESPONSE Réponse complète de l'agent
TOOL_START Appel d'outil avec paramètres
TOOL_OK Résultat de l'outil (succès ou erreur)
LLM_USAGE Consommation de tokens, modèle, latence

Protection des données UE

RGPD / GDPR

AIMOS respecte le RGPD non par ajustement après coup, mais par des décisions architecturales — Privacy by Design.

Art. 25 — Protection des données dès la conception

  • Traitement des données local par défaut (SovereignNode)
  • PII-Vault anonymise avant toute transmission externe
  • Aucun service cloud dans la configuration standard
  • Minimisation des données : seules les données contextuelles pertinentes dans le prompt LLM

Art. 32 — Sécurité du traitement

  • Chiffrement : PostgreSQL SSL, SQLite on-disk
  • Contrôle d'accès par Execution Rings (3 niveaux)
  • Piste d'audit complète de toutes les opérations de traitement
  • Mappings PII liés à la session (pas de stockage PII persistant)

Conformité aux normes

ISO 9001 & 27001

AIMOS soutient les exigences des deux normes grâce à des mécanismes intégrés.

ISO 9001 — Management de la qualité

  • Traçabilité : chaque décision dans le journal d'audit
  • Rétro-traçabilité : les identifiants de session relient tous les événements
  • Processus documentés : System-Prompts comme procédures opérationnelles
  • Actions correctives : le cycle Dreaming apprend des erreurs

ISO 27001 — Sécurité de l'information

  • Classification : les Execution Rings définissent les niveaux d'accès
  • Contrôle d'accès : activation des outils configurable par agent
  • Journalisation des événements : les 6 types d'événements sont capturés
  • Sécurité physique : les données ne quittent pas le SovereignNode

Comptabilité

Conformité GoBD

Les GoBD (principes de tenue et de conservation ordonnées des livres, registres et documents sous forme électronique) imposent des exigences particulières aux systèmes assistés par IA.

AIMOS résout cela par une séparation architecturale stricte :

Séparation : Brouillon vs. Validation

Les agents préparent les écritures comptables (brouillon). La validation est effectuée exclusivement par des comptables autorisés dans le système cible (ETA V8, DATEV, etc.). Aucun LLM n'a de droits transactionnels.

Human-in-the-Loop

Les actions Ring 2 (accès en écriture) nécessitent une activation explicite. Les opérations comptables requièrent toujours une approbation humaine — imposé par l'architecture, pas par une simple politique.

En vigueur à partir d'août 2026

Réglementation européenne

Règlement IA de l'UE (Loi sur l'IA)

Le Règlement IA (UE) 2024/1689 entre progressivement en vigueur et concerne toute entreprise déployant des systèmes d'IA. AIMOS est conçu architecturalement pour la conformité — pas adapté après coup.

Classification des risques : Risque limité

Les agents AIMOS sont des systèmes d'assistance avec surveillance humaine (Human-in-the-Loop). Aucune décision autonome concernant des personnes, aucune surveillance biométrique, aucune fonction de notation sociale. Ils relèvent ainsi de la catégorie « risque limité » — et non « haut risque ».

Obligation de transparence : Respectée

Art. 50 : Les utilisateurs doivent savoir qu'ils interagissent avec une IA. Chaque agent AIMOS s'identifie comme assistant IA dans son prompt système. Toutes les actions sont traçables dans la piste d'audit.

Surveillance humaine : Imposée par l'architecture

Art. 14 : Les Execution Rings garantissent qu'aucun agent n'exécute d'actions critiques de manière autonome. Écritures comptables, e-mails et commandes sont présentés comme brouillons — c'est l'humain qui décide.

Documentation technique : Complète

Art. 11 : Architecture système, flux de données, description du modèle, indicateurs de performance et évaluation des risques sont documentés et consultables. Licence MIT : accès intégral au code source.

Journalisation : Exhaustive

Art. 12 : Chaque action d'agent est journalisée — appels d'outils, requêtes LLM, consommation de tokens, chaînes de décision. Pistes d'audit infalsifiables avec identifiants de session et horodatages.

Souveraineté des données : Complète

Art. 10 : Les données d'entraînement et de traitement ne quittent jamais le SovereignNode. Le PII-Vault anonymise automatiquement les appels API externes. Aucune dépendance au cloud.

Avantage de l'IA locale

Les services d'IA cloud doivent respecter des obligations plus strictes en tant qu'« IA à usage général » (GPAI). AIMOS utilise un modèle open source exploité localement — les obligations GPAI incombent au fournisseur du modèle (Alibaba/Qwen), pas à l'opérateur.
