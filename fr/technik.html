Infrastructure

SovereignNode

Un seul serveur. GPU locale. Aucune dépendance cloud. Le SovereignNode est le coeur de chaque installation AIMOS — un serveur physique ou virtuel qui héberge tous les composants.

Tout fonctionne on-premise : l'inférence LLM, les bases de données, les processus agents et les canaux de communication. Aucun octet ne quitte votre réseau — sauf si vous le configurez explicitement (par ex. messages Telegram).

Inférence

Intégration LLM

Inférence locale via SGLang. Fonctionnement séquentiel. Gestion intelligente de la VRAM.

Inférence LLM locale (SGLang)

AIMOS s'appuie sur un modèle de 27 milliards de paramètres avec Tool-Calling natif. Les modèles plus petits (<20B) échouent à l'appel d'outils fiable — un résultat critique de notre évaluation en production. RadixAttention : le prefix cache est partagé entre les agents — aucun rechargement lors du changement d'agent.

Fonctionnement séquentiel

Le VRAM Guard garantit qu'un seul agent accède à la GPU à la fois. Les requêtes sont maintenues dans la file d'attente de la base de données et traitées séquentiellement — pas d'OOM, pas de conflit VRAM.

Stratégie Keep-Alive

Le modèle reste 30 minutes en VRAM. Tous les agents partagent le même modèle — pas de déchargement lors du changement d'agent. La VRAM n'est libérée qu'après 30 minutes d'inactivité.

Dual-DB

Architecture base de données

AIMOS utilise deux systèmes de bases de données avec des responsabilités clairement séparées :

PostgreSQL (base de données relais)

Relais central de messages entre le Shared Listener, l'orchestrateur et les agents. Stocke les messages entrants, les journaux d'audit, les mappings PII-Vault et les données de session. Multi-processus grâce au Connection Pooling.

SQLite (Agent-Memory)

Chaque agent possède sa propre base de données SQLite avec mémoire sémantique, épisodique et procédurale. Recherche hybride via FTS5 + embeddings vectoriels. Portable par simple copie du fichier.

Points forts techniques

Ce qui distingue AIMOS

Tool-Calling natif

Pas de hacks textuels ni de parsing regex — AIMOS utilise l'API de Tool-Calling native du LLM. L'agent pilote directement les systèmes, au lieu de simplement décrire les actions.

Voix multilingue

Reconnaissance vocale (Whisper STT) et synthèse vocale (Piper TTS) dans toutes les langues — les agents comprennent les messages vocaux et répondent dans la langue maternelle de l'utilisateur.

Suivi des tokens

Chaque appel LLM est enregistré : tokens entrée/sortie, latence, utilisation du contexte. Transparence totale des coûts par agent, par conversation, par mois.

Conversation Threading

Chaque agent sait avec qui il parle sur quel canal. Telegram, e-mail et messages internes sont proprement séparés — aucune confusion entre les interlocuteurs.

Interopérabilité

Portabilité des agents

Les agents AIMOS sont portables, compatibles et interopérables grâce aux standards ouverts.

Export/Import OAP

Le format Open Agent Package permet l'export complet d'un agent incluant Memory, Skills et configuration sous forme d'archive portable.

agent_export.oap
  config.yaml
  memory.sqlite
  skills/
  prompts/

MCP Bridge (39 outils)

Le Model Context Protocol permet aux LLM externes (Claude, GPT, etc.) d'accéder aux Skills AIMOS. 39 outils sont disponibles en tant que serveur MCP.

sql_query file_read rest_call memory_search +35 de plus

A2A Agent Cards

Chaque agent publie une Agent Card (JSON-LD) selon la spécification Google A2A. Les systèmes externes peuvent interroger les capacités, formats d'entrée et niveau de confiance.

"name": "Agent Construction",
"skills": ["cad_read", "bom_gen"],
"trust_ring": 1
